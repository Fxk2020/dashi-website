## 代码审查的重要性

Code Review 不仅是发现 Bug 的过程，更是团队知识共享和代码质量提升的重要环节。

## 审查者的角色

### 1. 关注点

- **功能实现**：代码是否实现了需求
- **代码质量**：可读性、可维护性
- **性能问题**：是否有明显的性能瓶颈
- **安全隐患**：是否存在安全漏洞

### 2. 审查技巧

#### 先看大局
了解 PR 的目的和整体设计思路。

#### 逐行细看
注意变量命名、逻辑分支、边界条件。

#### 提出建设性意见
不要只说"这里不好"，要说明为什么不好，如何改进。

## 被审查者的准备

### 1. 提交前自查

- 运行所有测试
- 检查代码格式
- 补充必要的注释

### 2. 写好 PR 描述

清晰说明：
- 解决了什么问题
- 如何解决的
- 有哪些注意事项

### 3. 及时响应反馈

积极回应审查意见，虚心接受建议。

## 团队规范建议

1. **设置 Review 时间上限**：避免 PR 积压
2. **使用 Checklist**：确保审查的完整性
3. **自动化工具辅助**：Lint、测试覆盖率等

## 常见问题

### 如何处理分歧？
通过讨论达成共识，必要时寻求技术负责人的意见。

### 审查时间太长怎么办？
拆分大 PR，使用增量审查。

## 总结

好的 Code Review 文化需要团队共同建设，既要严格把关，也要保持友好和建设性。
